# Community Supported Software - Local Automation Demo

This project demonstrates that multiple Community Supported Software systems can run together locally with automated cross-system workflows. All systems run via Docker for security and portability.

## Golden Rules (Strict Adherence Required)
1. **Schema:** For Digital Objects, use `type="summary"`. NEVER use `scopecontent`.
2. **File Handling:** Image format strings must be `jpeg` (not `jpg`) for API validation.
3. **Pathing:** Always use Windows backslashes `\` or Python raw strings `r"C:\..."`.
4. **Safety:** Never delete data without confirmation. "AI as Draft, Human as Editor."

## Image Automation Pipeline

### Quick Start Command
User says: "Run the full automation with prefix 'Demo - '"

**Single command (recommended):**
```bash
python import_to_all.py staging_images --prefix "Demo - " --caption
```

This executes:
1. Pre-flight check - Verifies all 5 systems are responding
2. AI captioning - Sends images to LM Studio, outputs captions.csv
3. Import - Creates records in all 5 systems with cross-links
4. DSpace thumbnails - Generated automatically at the end

**After Docker restarts:** Run `python start_demo.py` first to ensure all services are ready.

### Command-Line Options
```
python import_to_all.py <folder> [options]

Options:
  --prefix, -p    Prefix for titles (e.g., 'Demo - ')
  --caption, -c   Run AI captioning first (requires LM Studio)
  --skip-preflight  Skip pre-flight service check
```

### Scripts (in C:\Archivespace\Automation\)
- `start_demo.py` - **Run after Docker restarts** to fix stale port bindings and verify all services
- `import_to_all.py` - **Main script** - Five-system import with pre-flight check, optional captioning (--caption), and retry logic
- `caption_folder.py` - AI image captioning via LM Studio (can be run standalone or via --caption flag)
- `import_to_both.py` - Legacy dual-system import (DSpace + ArchivesSpace)
- `import_to_trio.py` - Legacy three-system import (adds VIVO)
- `explore_vivo.py` - VIVO API exploration and testing script
- `dspace_delete_community.py` - Cleanup utility (accepts --prefix)
- `captions.csv` - Intermediate data file (generated by captioning)
- `CST LOGINS.txt` - All system credentials in one file

### Cleanup Commands
- Delete single community: `python dspace_delete_community.py --prefix "Demo - "`
- Delete ALL communities: Use inline Python script (see below)

```python
# Delete all DSpace communities (run from C:\Archivespace\Automation)
python -c "
import requests
session = requests.Session()
API = 'http://localhost:8081/server/api'
session.get(API)
r = session.get(f'{API}/security/csrf')
csrf = session.cookies.get('DSPACE-XSRF-COOKIE') or r.json().get('token')
r = session.post(f'{API}/authn/login', data={'user': 'YOUR_EMAIL', 'password': 'YOUR_PASSWORD'}, headers={'Content-Type': 'application/x-www-form-urlencoded', 'X-XSRF-TOKEN': csrf})
auth = r.headers.get('Authorization')
csrf = session.cookies.get('DSPACE-XSRF-COOKIE')
r = session.get(f'{API}/core/communities', headers={'Authorization': auth})
for c in r.json().get('_embedded', {}).get('communities', []):
    csrf = session.cookies.get('DSPACE-XSRF-COOKIE')
    session.delete(f'{API}/core/communities/{c[\"uuid\"]}', headers={'Authorization': auth, 'X-XSRF-TOKEN': csrf})
    print(f'Deleted: {c[\"name\"]}')
"
```

## Cross-System Linking (All 5 Systems)

The automation creates bidirectional links connecting records across all systems:

### ArchivesSpace Links To:
| System | Link Type | Location |
|--------|-----------|----------|
| DSpace | File Version + External Document | Digital Object |
| VIVO | External Document | "Link to VIVO Research Record" |
| Fedora | External Document | "Link to Fedora Repository" |

### DSpace Links To:
| System | Metadata Field | Description |
|--------|----------------|-------------|
| ArchivesSpace | `datacite.relation.isReferencedBy` | Clickable link on item page |
| Fedora | `dc.relation` | Fedora resource URI |

### VIVO Links To:
| System | Link Type | Description |
|--------|-----------|-------------|
| DSpace | URLLink | "View in DSpace" |
| ArchivesSpace | URLLink | "View in ArchivesSpace" |
| Fedora | URLLink | "View in Fedora Repository" |

### Fedora Links To:
| System | Property | Description |
|--------|----------|-------------|
| DSpace | `dcterms:relation` | DSpace item URL |
| ArchivesSpace | `rdfs:seeAlso` | ArchivesSpace staff URL |
| VIVO | `rdfs:seeAlso` | VIVO dataset URL |

### CollectionSpace Links To:
| System | Field | Description |
|--------|-------|-------------|
| All Systems | Comments | URLs to DSpace, ArchivesSpace, VIVO, and Fedora |

**Note:** The `datacite.relation.isReferencedBy` field is used in DSpace (not `dc.relation`) because DSpace's Angular frontend renders it as a clickable hyperlink.

**ArchivesSpace URL Format:** Links use the resolve endpoint for proper routing:
`http://localhost/staff/resolve/readonly?uri=/repositories/2/digital_objects/XXX`

## VIVO Integration

### Docker Setup (C:\Archivespace\vivo\)
```yaml
services:
  vivo-solr:
    container_name: vivo-solr
    image: vivoweb/vivo-solr:latest
    ports:
      - 8985:8983
    healthcheck:
      test: ["CMD-SHELL", "wget --no-verbose --tries=1 --spider http://localhost:8983/solr/vivocore/admin/ping || exit 1"]
      interval: 15s

  vivo:
    container_name: vivo
    image: vivoweb/vivo:latest
    environment:
      - SOLR_URL=http://vivo-solr:8983/solr/vivocore
    ports:
      - 8082:8080
    depends_on:
      vivo-solr:
        condition: service_healthy
```

### VIVO API Details
- **SPARQL Query Endpoint:** `http://localhost:8082/api/sparqlQuery`
- **SPARQL Update Endpoint:** `http://localhost:8082/api/sparqlUpdate`
- **Default Graph:** `http://vitro.mannlib.cornell.edu/default/vitro-kb-2`
- **Namespace for new entities:** `http://vivo.mydomain.edu/individual/`

### VIVO Image Upload Process
VIVO does NOT have an API for image uploads. Images must be uploaded via web forms:

1. **Navigate to add page:** `GET /uploadImages?entityUri={uri}&action=add`
2. **Upload image:** `POST /uploadImages?entityUri={uri}&action=upload` with multipart form
3. **Submit crop:** `POST /uploadImages?entityUri={uri}&action=save` with x, y, w, h coordinates

**Critical:** Crop coordinates must match actual image dimensions. Use PIL to get dimensions:
```python
from PIL import Image
img = Image.open(image_path)
img_width, img_height = img.size
crop_size = min(img_width, img_height)  # Square crop
x_offset = (img_width - crop_size) // 2  # Centered
y_offset = (img_height - crop_size) // 2
```

### VIVO Entity Types Used
- `foaf:Person` - For photographer/creator records
- `vivo:Dataset` - For image/collection records with cross-system links

### Cross-System Links in VIVO
Datasets include `vivo:URLLink` objects pointing to:
- DSpace item URL
- ArchivesSpace digital object URL

- **Authentication:** Basic Auth (see config.py for credentials)
- **Data Format:** JSON-LD (Linked Data)

### Quick API Examples
```bash
# Get root container (JSON-LD)
curl -u $FEDORA_USER:$FEDORA_PASS -H "Accept: application/ld+json" http://localhost:8083/fcrepo/rest/

# Create a new container
curl -u $FEDORA_USER:$FEDORA_PASS -X POST http://localhost:8083/fcrepo/rest/ -H "Content-Type: text/turtle" -d "<> <http://purl.org/dc/elements/1.1/title> \"Test Object\" ."

# Upload a binary (file)
curl -u $FEDORA_USER:$FEDORA_PASS -X POST http://localhost:8083/fcrepo/rest/ -H "Content-Type: image/jpeg" --data-binary @image.jpg
```

## CollectionSpace Integration

### Docker Setup (C:\Archivespace\collectionspace\)
CollectionSpace runs with three containers: PostgreSQL database, ElasticSearch, and the application server.

```yaml
services:
  db:
    container_name: cspace-db
    image: postgres:12
    environment:
      - POSTGRES_USER=csadmin
      - POSTGRES_PASSWORD=csadmin
    ports:
      - 5434:5432
    volumes:
      - cspace-db-data:/var/lib/postgresql/data

  es:
    container_name: cspace-es
    image: elasticsearch:7.17.9
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
    ports:
      - 9202:9200

  collectionspace:
    container_name: collectionspace
    image: collectionspace/dev:latest
    environment:
      - DB_HOST=db
      - DB_CSADMIN_PASSWORD=csadmin
      - ES_HOST=http://es:9200
    ports:
      - 8180:8180
    volumes:
      - cspace-services:/services
      - cspace-application:/application
      - cspace-tomcat:/apache-tomcat-8.5.51
    depends_on:
      - db
      - es
```

### CollectionSpace API Details
- **Web UI:** `http://localhost:8180/cspace/core/login`
- **REST API Root:** `http://localhost:8180/cspace-services/`
- **Authentication:** Basic Auth (see config.py for credentials)
- **Active Tenant:** core (ID: 1)

### Quick API Examples
```bash
# List all collection objects
curl -u $CSPACE_USER:$CSPACE_PASS "http://localhost:8180/cspace-services/collectionobjects"

# List all accounts
curl -u $CSPACE_USER:$CSPACE_PASS "http://localhost:8180/cspace-services/accounts"

curl -u $CSPACE_USER:$CSPACE_PASS -X POST \
  -H "Content-Type: application/xml" \
  "http://localhost:8180/cspace-services/collectionobjects" \

  <ns2:collectionobjects_common xmlns:ns2="http://collectionspace.org/services/collectionobject">
    <objectNumber>2024.1.1</objectNumber>
    <objectName>Test Object</objectName>
  </ns2:collectionobjects_common>
</document>'
```

## Troubleshooting

### After Docker Desktop Restarts
**IMPORTANT:** After restarting Docker Desktop, run the startup script to fix stale port bindings:
```bash
python start_demo.py
```
This restarts containers with stale ports (image-server, proxy, archivesspace), starts CollectionSpace Tomcat, and verifies all services are responding.

### Images Not Appearing in ArchivesSpace
The image-server container often has stale port bindings after Docker restarts. Fix with:
```bash
docker restart image-server
```

### DSpace Thumbnails Not Appearing
Run: `docker exec dspace //dspace/bin/dspace filter-media`
Note: Use `//dspace` (double slash) to prevent Git Bash path translation on Windows.

### VIVO Solr Connection Error (500)
If VIVO shows 500 errors, check that `runtime.properties` uses Docker network hostname:
- Correct: `vitro.local.solr.url = http://vivo-solr:8983/solr/vivocore`
- Wrong: `vitro.local.solr.url = http://localhost:8983/solr/vivocore`

### VIVO Images Show Wrong Section
Crop coordinates must match actual image dimensions. If images show tiny corner sections, the crop x/y/w/h values don't match the uploaded image size.

### CollectionSpace Tomcat Not Starting
If CollectionSpace doesn't respond, start Tomcat manually:
```bash
docker exec -e DB_HOST=db -e DB_CSADMIN_PASSWORD=csadmin collectionspace sh -c "cd /apache-tomcat-8.5.51 && ./bin/startup.sh"
```

### CollectionSpace Build Failed
If you need to rebuild CollectionSpace:
```bash
# Inside the collectionspace container
cd /services && ant deploy -Djee.dir=/apache-tomcat-8.5.51
cd /services && ant import -Djee.dir=/apache-tomcat-8.5.51
```

### Checking CollectionSpace Logs
```bash
docker exec collectionspace tail -50 /apache-tomcat-8.5.51/logs/catalina.out
```
